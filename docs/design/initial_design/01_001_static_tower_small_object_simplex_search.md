# Static Tower Small-Object Simplex Search

## Status

Design note from the Project Owner discussion on June 12, 2026.

This document records the intended first package shape: a static quotient-tower
simplex search for a sparse graph, including degenerate simplices. The Kan
replacement and cofibrant/small-object replacement variants are future design
tracks; this note describes the small-object/static-tower version only.

## Attribution

The algorithm recorded here is PM Abdul Malik's work and part of his thesis.
This includes the static quotient-tower pipeline, the degree-wise `Out`
construction, the treatment of degenerate simplices, and the fiber-addressed
lift search.

The Project Owner relayed and clarified the algorithm in the June 12, 2026
design discussion and corrected this document's attribution: the whole algorithm
should be attributed to Abdul Malik, not only the lift-search speedup.

Codex's role in this document is consultant synthesis: organizing the discussion

## Goal

Given a graph `G` and a positive integer `k`, find all simplices of degree
`0 <= m <= k`, including degenerate simplices, using a quotient tower to avoid
unnecessary upstairs search.

The intended input graph is preferably sparse. The main speedups are:

- degree-wise sparse extension by cached `Out` frontiers;
- static quotient-tower search;
- fiber-addressed lifting from downstairs simplices, instead of post hoc
  projection checks.

## Graph Normalization

The first version normalizes loops before building the tower.

1. Strip loops from the input graph if loops are present.
2. Add exactly one formal identity loop at every node.
3. Treat the normalized graph as having `#Aut(s) = 1` for every node `s`.

This gives a single canonical identity arrow `id_s` at each vertex. Degenerate
simplices are then generated by the same extension rule as nondegenerate
simplices, rather than by a separate after-the-fact degeneracy pass.

The implementation should still distinguish formal identity arrows from original
non-loop graph arrows. If future variants need to preserve meaningful input
loops as non-identity arrows, that should be a separate design decision.

## Static Tower Pipeline

The first package pipeline is:

1. Normalize `G` so every vertex has exactly one formal identity loop.
2. Use a contraction schema to build a static quotient tower `G^bullet`.
3. Let `G^ell` denote the bottommost nondegenerate tier of `G^bullet`: `G^ell`
   is not `pi_0`, and the next tier is either `pi_0` or absent.
4. Directly enumerate all `m`-simplices of `G^ell` for `0 <= m <= k`, including
   degenerate simplices, using the degree-wise `Out` induction.
5. Inductively descend the tower. Once all simplices up to degree `k` are known
   at tier `r + 1`, search tier `r` only inside fibers over corresponding
   simplices downstairs.

The tower is static. This is not a dynamic exploration runtime.

## Degree-Wise Out Construction

Let `A(s)` be the formal-reflexive one-step target set:

```text
A(s) = {s} union {t : s -> t is a non-loop edge of G}
```

Equivalently, the one-step arrows out of `s` are:

```text
Out^1(s) = {id_s} union Out_G(s)
```

The simplex condition is a directed flag condition, not merely a path
condition. A degree-2 string:

```text
0 -> 1 -> 2
```

is a 2-simplex only if the actual face edge:

```text
0 -> 2
```

also exists. More generally, an `m`-simplex has edge witnesses for every
directed face edge `s_i -> s_j` with `i < j`, using `id_s` when `s_i = s_j`.
The induction below is the sparse way to check only the new face edges required
when a target is appended.

Define:

```text
Out^0(s) = {s}
Out^m(s) = {m-simplices with initial vertex s}
```

For an `m`-simplex

```text
sigma = (s_0, ..., s_m)
```

define its extension target frontier:

```text
F(sigma) = A(s_0) cap ... cap A(s_m)
```

Then every target `t in F(sigma)` gives an `(m + 1)`-simplex:

```text
sigma * t = (s_0, ..., s_m, t)
```

With `Out(sigma)` interpreted as the set of one-step simplex extensions of
`sigma`, the degree recursion is:

```text
Out^(m+1)(s) = union_{sigma in Out^m(s)} Out(sigma)
```

If `Out(sigma)` is interpreted as target vertices instead, then:

```text
Out^(m+1)(s) =
  {sigma * t : sigma in Out^m(s), t in F(sigma)}
```

## Cached Frontier Induction

The frontier should be an inductive payload carried by each simplex.

Let:

```text
partial_m(sigma) = (s_0, ..., s_{m-1})
tgt(sigma)       = s_m
```

Then:

```text
F(sigma)
  = F(partial_m(sigma)) cap A(tgt(sigma))
```

This avoids recomputing an intersection over every vertex of `sigma`. For sparse
graphs, extension mostly becomes:

```text
intersect cached prefix frontier with one adjacency/identity target set
```

A discovered simplex should therefore carry or reference:

```text
vertices / address
degree
face-edge witnesses
degeneracy data
projection data
cached extension frontier
```

The frontier is not merely a helper. It is part of the sparse dynamic-programming
state of the simplex search.

This is the speedup for the ordinary simplex condition. Extending `(0, 1)` by
`2` still requires `0 -> 2`; the requirement is enforced by checking whether `2`
lies in the cached frontier:

```text
F(0, 1) = A(0) cap A(1)
```

So the algorithm does not skip the long face edge. It finds it through a cached
intersection rather than by rebuilding all face checks from scratch.

## Degenerate Simplices

Degenerate simplices are first-class simplices.

For example:

```text
s -> s -> s -> s'
```

is a degree-3 simplex candidate:

```text
(s, s, s, s')
```

It is not the edge `s -> s'` promoted to degree 3. Its arity is part of its
identity for this search.

The formal identity loops make these strings appear naturally:

```text
(s)
extend by id_s           -> (s, s)
extend by id_s           -> (s, s, s)
extend by original s->s' -> (s, s, s, s')
```

Repeated vertices may leave the frontier unchanged:

```text
F(s, s)    = A(s) cap A(s) = A(s)
F(s, s, s) = A(s)
```

but the degenerate simplex record must not be collapsed merely because its
frontier agrees with a lower-dimensional spine.

This is essential for tower lifting. A nondegenerate upstairs simplex can
project to a degenerate downstairs simplex. For example, if `a`, `b`, and `c`
all project to `S`, while `d` projects to `S'`, then:

```text
(a, b, c, d) upstairs projects to (S, S, S, S') downstairs
```

The downstairs degree-3 degenerate simplex is the correct address for the
upstairs degree-3 simplex. Reducing it to `(S, S')` would destroy the
dimension-preserving lift.

For the first implementation, a degenerate simplex is identified by its realized
degree, vertex address, and face-edge witnesses. The package does not need to
emit multiple records for different formal histories of degeneracy operators
that produce the same realized simplex. In simplicial-set language, a repeated
vertex tuple may admit more than one way to describe how identities were
inserted, but that provenance is not part of the first small-object enumerator
unless a later Kan or proof-producing mode explicitly needs it.

## Fiber-Addressed Tower Lifting

Let:

```text
p_r : G^r -> G^(r+1)
```

be the projection from tier `r` to the next downstairs tier `r + 1`.

Suppose a downstairs `(m + 1)`-simplex at tier `r + 1` is known:

```text
tau = (c_0, ..., c_m, c_(m+1))
```

Write:

```text
partial tau = (c_0, ..., c_m)
alpha       = last edge of tau, c_m -> c_(m+1)
```

At tier `r`, the search is not:

```text
enumerate arbitrary upstairs extensions, then check whether they project to tau
```

Instead, `tau` is the address of the search. For each upstairs `m`-simplex
`sigma` already known to lie over `partial tau`, only inspect upstairs edges
lying over `alpha`.

If:

```text
sigma = (s_0, ..., s_m)
tgt(sigma) = s_m
```

then the candidate targets are:

```text
LiftTargets_r(alpha, s_m) cap F(sigma)
```

where:

```text
LiftTargets_r(alpha, s_m)
  = {t : there is an upstairs edge beta : s_m -> t with p_r(beta) = alpha}
```

Each such target emits:

```text
sigma * t
```

as an upstairs `(m + 1)`-simplex over `tau`.

Equivalently:

```text
Fib_r(tau)
  =
  {sigma * beta :
     sigma in Fib_r(partial tau),
     beta starts at tgt(sigma),
     p_r(beta) = alpha,
     target(beta) in F(sigma)}
```

The check that the extra edge is the final edge of a downstairs `(m + 1)`-simplex
is built into candidate generation. It is not an ad hoc validation step.

## Why This Is The Speedup

The naive upstairs search would consider many candidate extensions and then
project them downstairs to see if their projection is a known simplex.

The intended search never enters those irrelevant regions. It is indexed by:

```text
downstairs simplex tau
downstairs final edge alpha
upstairs fiber over partial tau
upstairs edge fiber over alpha from the current target
upstairs frontier F(sigma)
```

So the extension step is:

```text
targets =
  edge_fiber[alpha][tgt(sigma)]
  cap frontier[sigma]
```

This is Abdul Malik's lift-speedup insight in implementation form: the
downstairs simplex and its final edge determine where search is allowed to
happen in the first place.

## Degenerate Downstairs Fibers

This fiber-addressed search also handles degenerate projections correctly.

If the downstairs final edge is:

```text
id_C : C -> C
```

then its upstairs edge fiber can include formal identities:

```text
id_s : s -> s
```

for `s` over `C`, and may also include non-identity upstairs internal edges:

```text
s -> s'
```

with both endpoints projecting to `C`.

This is how nondegenerate upstairs simplices can lie over degenerate downstairs
simplices without ever searching for a larger-dimensional object over a
smaller-dimensional object. Dimension is preserved:

```text
upstairs (m + 1)-simplex over downstairs (m + 1)-simplex
```

The downstairs simplex may simply be degenerate.

## Relation To state_collapser

The intended dependency on `state_collapser` is structural:

- use the package's graph, edge, state, and contraction/tower concepts;
- build a static quotient tower from a complete graph snapshot;
- treat the relevant `state_collapser` objects as static after initialization;
- use tier projections, fibers, and action/edge support style queries;
- do not use the dynamic `TowerRuntime` exploration loop as the core algorithm;
- do not use RL training as the core algorithm.

This package must not reinvent machinery that already exists in
`state_collapser`. The quotient tower, contraction schema handling, stable
state/action/edge identities, tier projections, cell membership, and edge
source/target relationships should come from `state_collapser` objects.

The intended first integration shape is:

```text
input graph
  -> state_collapser State / PrimitiveAction / BaseEdge records
  -> state_collapser contraction schema
  -> state_collapser static PartitionTower
  -> this package's simplex indexes and simplex-fiber search
```

This package may build simplex-specific indexes over a static tower, but those
indexes are derived caches, not a second implementation of the tower. Examples
of appropriate local indexes are:

```text
formal identity arrows for simplex degeneracy
frontier F(sigma) caches
simplex records and simplex ids
simplex fibers over downstairs simplex ids
edge-fiber target lookup tables derived from tower edge/cell data
artifact/reporting tables
```

Examples of things this package should not duplicate are:

```text
state and edge registries
contraction schema execution
tier construction
state-cell membership
projection between tiers
quotient edge/action-cell ownership
source/target information for tower edges
```

If a required static query is not exposed cleanly by `state_collapser`, the
preferred path is to add or wrap the missing `state_collapser` surface rather
than recreate its semantics in this package.

The analogy to `state_collapser` training surfaces is:

```text
FrozenQuotientBehavior -> known downstairs simplex
PathFiber              -> simplex fiber over that downstairs simplex
action mask/frontier   -> F(sigma) and edge-fiber target intersection
training step          -> enumerated lift or diagnostic miss
```

But the package being designed here is an enumerator, not a learner.

One important implementation caution: control-oriented tower logic may suppress
internal loops from live action surfaces. This simplex engine must retain the
edge-fiber meaning of internal edges over downstairs identities, because those
edges are exactly where nondegenerate upstairs simplices can sit over degenerate
downstairs simplices.

## Input Loops

The first small-object version strips input loops and then adds one formal
identity loop at each vertex. This is safe only for the intended first-scope
semantics, where loop edges in the input are not being treated as distinct
non-identity arrows.

If a future graph format needs meaningful non-identity loops, those loops should
not be discarded. They should be represented as ordinary loop arrows alongside
the formal identity:

```text
id_s                 formal degeneracy arrow
lambda : s -> s      ordinary non-identity input loop
```

That is a separate design track. The first version uses exactly one loop per
vertex so degeneracy has a canonical identity witness.

## Kan Variant Contrast

The future Kan version is a different search problem. The small-object/static
tower algorithm in this note enumerates full directed-flag simplices: a candidate
simplex must have all required face edges.

The Kan version should instead be understood as inner-horn filling. There the
search input is a composable path or horn, and the algorithm searches for
simplex fillers. In that variant, the natural objects to enumerate first are
length-`m` paths for `0 <= m <= k`, then fillers over those paths/horns.

This note does not specify the Kan filling contract. It only records the contrast
so the first implementation does not accidentally conflate:

```text
small-object version: enumerate existing full simplices
Kan version: enumerate paths/horns and search for fillers
```

## First-Scope Artifact Format

The first implementation should produce a machine-readable artifact that records
what was searched and what was found. This matters because the output can be too
large for a prose log, and because the quotient-tower speedup depends on
projection and fiber bookkeeping that should be auditable.

The source artifact should be JSON or JSONL tables, with a human-readable report
derived from that source when needed. A first-scope artifact should include:

```text
manifest
  package versions
  input graph id or fingerprint
  k
  contraction schema name/config
  loop normalization policy
  state_collapser tower fingerprint or equivalent summary

tier_summary
  tier ids
  vertex/state-cell counts
  edge/action-cell counts
  pi_0 status
  selected bottommost nondegenerate tier G^ell

simplex_records
  simplex id
  tier
  degree
  vertex/cell address
  face-edge witnesses
  initial vertex
  target vertex
  prefix simplex id
  last edge id
  degenerate flag
  projection simplex id, when present
  frontier count, with optional frontier members

simplex_fibers
  downstairs simplex id
  upstairs tier
  upstairs simplex ids or count/shard reference

edge_fibers
  downstairs edge id
  upstairs source id
  upstairs target ids or count/shard reference

diagnostics
  simplex counts by tier and degree
  degenerate/nondegenerate counts
  fiber sizes
  frontier intersection counts
  timing and memory summaries when available
```

For small graphs, the artifact may inline full simplex and fiber membership. For
large graphs, it should support sharded records or count-only summaries with
stable ids. The key requirement is that every reported simplex can be traced
back to:

```text
its tier
its vertex/cell address
its face-edge witnesses
its projection downstairs
the downstairs simplex fiber that caused it to be searched
```

The artifact format is not meant to replace `state_collapser` internals. It is a
readout of this package's simplex enumeration over a static `state_collapser`
tower.

## First Implementation Commitments

The following are implementation commitments, not unresolved mathematical
questions for the Project Owner.

### state_collapser Boundary

Engineers should implement the thinnest practical adapter over static
`state_collapser` tower APIs. The Project Owner should not need to choose exact
import paths or wrapper method names unless the existing `state_collapser`
surface is missing a mathematically required query.

The adapter should name the queries this simplex search needs, such as:

```text
tiers
state/cell members
edge members
source and target of each edge
projection of a vertex/cell/edge from tier r to tier r+1
edge fibers over a downstairs edge
identity/formal-loop handling at each tier
```

That boundary should be a thin adapter or wrapper around `state_collapser`
objects, not a new tower implementation. Its purpose is to keep this package's
simplex code independent from incidental `state_collapser` internals while
still reusing the actual `state_collapser` tower, registry, schema, and
projection semantics.

There is no additional mathematical concern here. The engineering rule is
simple:

```text
state_collapser owns the tower.
This package owns simplex enumeration over that tower.
```

The adapter exists only to keep that boundary tidy in code. It should translate
from `state_collapser`'s static tower objects into the simplex search's local
lookup shape. It must not reinterpret contractions, projections, fibers, edge
ownership, or source/target relations.

If implementation discovers that a needed static tower relation is awkward to
query, that is an engineering integration issue, not a new PO design question.
The default response is to write the smallest wrapper over the existing
`state_collapser` object graph and continue.

### Artifact Layout

The accepted direction is:

- use a compact single-file artifact for tiny examples and unit tests;
- use a manifest-plus-table layout for serious runs;
- allow JSONL shards for large simplex/fiber tables.

Simplex enumeration can produce many records. A single JSON file is convenient
for tests, but it becomes awkward when simplex and fiber tables are large. JSONL
shards are better for large outputs because records can be streamed and
inspected incrementally. The first serious artifact format should therefore be:

```text
readout_source.json
  manifest
  tier summaries
  table paths
  counts and fingerprints

simplex_records.jsonl
simplex_fibers.jsonl
edge_fibers.jsonl
diagnostics.json
```

The exact first implementation can start with a compact single-file artifact for
small tests, but the design should leave a direct path to manifest-plus-table
artifacts so larger sparse graphs do not force a format rewrite.
